# Findings: Claude Skill Design for Declarative, Test-Linked, Drift-Tracked User Journeys

## Summary

The repo already contains two-thirds of the pattern this project needs, just not wired together. `ux-journey-mapper` proves the lean-agent-loop discovery pattern (parallel agents, `/tmp` full output, JSON summaries, single markdown artifact) but is explicitly one-shot — it has no persistent spec schema, no test linkage, and no re-run/diff semantics. `quality:test-planner` (and its more detailed cousin `docs:write`) show the two things `ux-journey-mapper` lacks: a repeatable "map code changes to coverage" analysis process, and gated, template-driven document authoring. Playwright's own `playwright-test-planner` agent shows the minimal viable version of "explore app → save structured markdown plan" but has no drift detection either — it's a generator, not a maintainer.

Anthropic's official skill-authoring guidance (fetched live from `platform.claude.com/docs/en/agents-and-tools/agent-skills/best-practices`) gives concrete, quotable rules: SKILL.md body under 500 lines, name/description frontmatter limits (64 chars / 1024 chars, third-person description, no reserved words), progressive disclosure one level deep, "workflows with checklists" and "validator → fix → repeat" feedback-loop patterns for exactly this kind of iterative maintenance task, and an explicit "plan-validate-execute" pattern for batch/destructive updates — directly applicable to "re-run to enrich journeys without clobbering hand-edited content."

**Recommendation**: build a **small family of 3 skills** (extract, verify/link, enrich-and-drift-check) that share one declarative markdown+frontmatter journey spec format (the sibling subtopic's job to define), composed the same way `ux-journey-mapper` → `quality:test-planner` → `docs:write` already compose informally in this repo. Do not build one monolithic skill; the phases have genuinely different re-run cadences (extract: rare/on-demand, verify: every CI run or pre-commit, enrich: periodic/opportunistic) and different freedom levels (extract = high freedom exploration, verify = low freedom deterministic script, enrich = medium freedom).

## Options Surveyed

1. **Single monolithic "journey-tracker" skill** with internal phase-branching (extract | verify | enrich as sub-workflows inside one SKILL.md), modeled loosely on `docs:write`'s single-file multi-phase-gate design.
2. **Three-skill family** — `journeys-extract` (discovery, lean-agent-loop, like `ux-journey-mapper`), `journeys-verify` (test-linkage + drift/staleness check, like `quality:test-planner`'s coverage-mapping process but pointed at journey files instead of code diffs), `journeys-enrich` (incremental re-run that fills gaps without destroying prior human edits, gated like `docs:write`).
3. **Two-skill split** — merge verify+enrich into one "maintain" skill (since both re-run against an existing spec) and keep extract separate (since it's the only phase that does open-ended codebase discovery).
4. **Skill + bundled script hybrid** — one skill, but push the deterministic parts (drift diffing, test-link validation) into a bundled Python/shell script per Anthropic's "low freedom → specific script" guidance, keeping only the high-freedom extraction/enrichment prose in SKILL.md.
5. **No new skill — extend `ux-journey-mapper` in place** to add a "re-run mode" that diffs against the existing `docs/ux/journey-map.md` and appends a "Linked Tests" column.

|---|---|---|---|---|
| 1. ## Risk and Failure Modes

- **Freedom-level mismatch inside one skill.** Anthropic's guidance is explicit that low-freedom (deterministic script), medium-freedom (template + parameters), and high-freedom (open exploration) content need different instruction styles. A monolithic journey skill will either over-constrain the exploratory extraction phase or under-constrain the deterministic verify phase. Strongest argument against Option 1.
- **Drift check confused with content quality check.** "Is this journey spec stale" (mechanical: does it reference files/tests/routes that still exist, has the linked code changed since last verified) differs from "is this journey spec any good" (semantic: are steps still accurate, missing edge cases). Conflating them risks either being too slow to run often (LLM judgment every time) or too shallow to catch real regressions (pure file-hash diffing misses flow changes that don't touch referenced files).
- **Re-run clobbering human edits.** `ux-journey-mapper` is one-shot and always overwrites `docs/ux/journey-map.md`. A journey-tracking skill re-run to "enrich" must NOT silently overwrite manually-added notes. Needs the "plan-validate-execute" pattern Anthropic documents (generate a diff/patch plan, get approval, then apply) rather than blind regeneration.
- **Test-link rot.** Linking journey steps to specific test file paths / test IDs is fragile — test files get renamed/split/merged independently of the journey doc. A brittle 1:1 path reference will go stale faster than the journey content itself (bounds what the verify skill can reliably check — string-match against a stable test ID is more durable than a file path).
- **Skill-selection ambiguity.** If the 3 skills have overlapping descriptions, Claude may load the wrong one or all three redundantly. Each needs a distinct trigger phrase set: extract = "no spec exists yet / rediscover flows from code"; verify = "check if journey specs are stale or test links are broken, e.g. in CI"; enrich = "improve/fill gaps in an existing spec without full rediscovery."
- **Cost of running extraction too often.** `ux-journey-mapper`'s own context budget table shows ~8-10k tokens even with the lean pattern (vs. 40-80k naive). If "enrich" is just "re-run extract and diff," every enrichment cycle pays this cost. Better: enrich should target only journeys flagged stale by the cheaper verify pass.

## Migration and Adoption Cost

- Reuses `lean-agent-loop` and the `ux-journey-mapper` parallel-agent pattern directly — no new orchestration primitive needed, only new prompts/schemas.
- `quality:test-planner`'s "Analysis Process" (identify affected areas → plan tests → document → implement) is a template that can be lifted almost directly for the verify skill's "identify affected journeys → check test links → report drift" process.
- `docs:write`'s phase-gate pattern (interview → outline → draft → gate → revise) is a good template for the enrich skill's human-approval checkpoint.
- Main new cost: designing and validating the declarative spec schema itself (owned by the sibling subtopic) and writing the deterministic verify script (test-ID cross-reference, staleness scoring) — genuinely new code, not just prompt composition.
- Naming: per Anthropic's gerund-form convention (`processing-pdfs`), vs. this repo's `{group}-{name}` convention (e.g. `ux-journey-mapper` itself breaks the gerund convention) — the planning phase should decide explicitly whether to follow Anthropic's stated preference or repo precedent.

## Operational Concerns

- **Where verify runs**: natural home is CI (pre-commit or PR check) — "does this diff touch a file/route referenced by a journey spec, and if so is the linked test still present and passing." Deterministic/scriptable per Anthropic's "solve, don't punt" guidance; should not require an LLM call for the common case (escalate to LLM only for ambiguous/semantic drift).
- **Where extract/enrich run**: on-demand, human-triggered (like `ux-journey-mapper` today) or slow periodic cadence (post-release, or when verify accumulates N stale flags) — not per-PR, given real agent cost.
- **Idempotency**: extraction must be safe to re-run without duplicating journeys already documented; needs a stable identity per journey (slug/ID in frontmatter) so re-runs upsert rather than append-duplicate.
- **Ownership boundary with the format subtopic**: verify skill's feasibility depends entirely on the spec schema carrying machine-checkable references (test IDs, file paths), not just prose links.
- **Multi-model testing**: per Anthropic's checklist, test each skill across Haiku/Sonnet/Opus — verify in particular (likely unattended/in CI) should be validated on a cheaper model to confirm it doesn't need Opus-level reasoning for the common case.

## Prior Art and Lessons Learned

- **`ux-journey-mapper`** (this repo, `/home/tstapler/dotfiles/.claude/skills/ux-journey-mapper/SKILL.md`): proves parallel lean-agent discovery + single markdown artifact output works well for one-shot mapping; explicitly does not solve persistence, drift, or test-linking. Its context-budget table (~8-10k vs. 40-80k naive) is a strong argument for keeping the lean-agent pattern in the extract skill.
- **`quality:test-planner`** (`/home/tstapler/dotfiles/.claude/skills/quality/skills/test-planner/SKILL.md`): short, high-freedom skill mapping code changes → test plan via a 3-step Analysis Process. Useful template for "map journey → linked tests," though it targets code diffs, not existing docs — the new verify skill needs an analogous but doc-first process.
- **`docs:write`** (`/home/tstapler/dotfiles/.claude/skills/docs/skills/write/SKILL.md`): demonstrates the phase-gate pattern and Diataxis type discipline. Directly reusable for the enrich skill's "propose changes → gate → apply" flow.
- **Playwright's `playwright-test-planner` agent** (bundled in `ui-playwright`'s pnpm deps): explore live app → design scenarios → save via a dedicated `planner_save_plan` tool rather than a generic file write. Worth considering a dedicated "save" tool/script for journey specs (validating frontmatter schema, deduping by slug) rather than freehand LLM writes.
- **`lean-agent-loop`** (`~/.claude/skills/lean-agent-loop/SKILL.md`): the "Ralph Wiggum principle" and two-layer output pattern (full output to `/tmp`, tiny JSON summary back) is load-bearing for a multi-journey verify pass — one lean agent per journey returning `{journey_id, status: stale|fresh, reasons: [...]}`.
- **Anthropic official best practices** (`platform.claude.com/docs/en/agents-and-tools/agent-skills/best-practices`, fetched live 2026-07-01): confirms SKILL.md frontmatter limits (name ≤64 chars, lowercase/hyphens only, no "anthropic"/"claude" reserved words; description ≤1024 chars, non-empty, third-person), 500-line SKILL.md body guidance, "one level deep" reference-file rule, the high/medium/low degrees-of-freedom framework (maps directly onto extract/enrich/verify), "workflow checklist" and "validator → fix → repeat" feedback-loop patterns (applicable to verify), and the "plan-validate-execute" pattern for batch/destructive changes (applicable to enrich, to avoid clobbering human edits). Also: "avoid time-sensitive info" and "build evaluations first."
- **`jaimefjorge/claude-acceptance-tests`** (GitHub, found via web search) [TRAINING_ONLY — verify]: "markdown-driven acceptance test framework powered by Claude Code + Playwright MCP," organizing tests as one numbered markdown file per user journey (`01-public-pages.md`, `02-auth-flow.md`) with `Action`/`Expected` step pairs Claude executes directly via Playwright MCP. Closest public example of "declarative markdown journey spec directly driving automated test execution" — but it treats the markdown *as* the test (Claude interprets/runs it live) rather than maintaining a spec *linked to* a separately-authored test suite. Different architecture, but directly relevant to the "markdown as executable spec" end of the design space.
- **`neonwatty/qa-skills`** and **`lackeyjb/playwright-skill`** (GitHub) [TRAINING_ONLY — verify]: broader Playwright/QA automation skill collections; snippets suggest workflow docs and multi-user-flow support but no confirmed staleness/drift-detection feature.
- **"CAI Drift Check"** (`mcpmarket.com/tools/skills/cai-drift-check`) [TRAINING_ONLY — verify — WebFetch returned HTTP 429, could not independently confirm]: search snippet describes a "diagnostic utility... scanning documents and comparing their metadata against Git commit history, categorizing them into a prioritized three-tier staleness system." If real, this is the closest conceptual prior art for the verify skill's mechanism (git-history-based staleness tiers rather than pure content diffing) — worth a follow-up fetch before finalizing the verify skill design.
- **`xiaolai/claude-agent-sdk-skill-autoupdated`** (GitHub, archived) [TRAINING_ONLY — verify]: self-updating documentation pipeline (npm/PyPI/GitHub change detection, daily research agents, deterministic post-update verification) for SDK docs specifically — not journeys, but reinforces separating deterministic verify from LLM-driven enrich.

## Open Questions

- Should the verify skill be implemented partly as a bundled script (per Anthropic's "prefer scripts for deterministic operations") rather than pure LLM instructions, given "does this test ID / file still exist" is fully deterministic? (Leaning yes — depends on the final spec schema from the sibling subtopic.)
- What identity/slug scheme will the spec format use for journeys, and does it support stable upsert on re-extraction? (Owned by the format subtopic, but blocks extract skill's idempotency design.)
- Should "enrich" be a fully separate skill, or a `--mode=enrich` flag/step inside the extract skill that diffs against an existing spec? Research leans toward separate skills for description/triggering clarity, but this is a real fork worth revisiting in planning.
- Is CAI Drift Check real and actively maintained, or should its git-history-based staleness-tier idea simply be reimplemented in-house? Needs a follow-up fetch (429 was likely transient) before treating it as confirmed prior art.
- Where should the three skills live in this repo's taxonomy — a new `journeys`/`journey` group (parallel to `ux`, `quality`, `docs`)?

## Recommendation

Build a **three-skill family** — e.g. `journeys-extracting`, `journeys-verifying`, `journeys-enriching` (or a repo-convention-matching naming) — rather than one monolithic skill. This mirrors the repo's existing informal chain (`ux-journey-mapper` discovery → `quality:test-planner` coverage-mapping → `docs:write` gated authoring) and aligns with Anthropic's own degrees-of-freedom guidance: extraction is high-freedom exploration (reuse `ux-journey-mapper`'s parallel lean-agent pattern almost directly), verification is low-freedom and should be largely a deterministic bundled script (test-ID/file-existence cross-referencing, cheap enough for CI, with the `lean-agent-loop` two-layer pattern reserved for the LLM-assisted semantic-drift portion only), and enrichment is medium-freedom template-filling gated by human approval before overwriting any spec file (borrowing `docs:write`'s phase-gate pattern). Keep verify decoupled and cheap enough to run in CI on every PR; keep extract and enrich as deliberately-triggered, higher-cost operations. Finalize this design only after the sibling format-landscape subtopic settles the spec schema, since the verify skill's machine-checkability depends entirely on what identifiers (test IDs, file paths, journey slugs) the format captures in frontmatter.
