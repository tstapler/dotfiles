## Concerns

- [ ] **Story 3.1.1 (cfgcaddy deployment)** — DEPLOYMENT CONFLICT: `.cfgcaddy.yml` already has `src: stapler-scripts/*` → `dest: bin/scripts` (line 21-23). Adding `src: stapler-scripts/git-ssh-fallback` → `dest: .local/bin/git` creates a double-link: the wildcard glob will also link the file under `~/bin/scripts/git-ssh-fallback`. On a machine where `~/bin/scripts` is in PATH before `~/.local/bin`, this creates a dangling collision — the wildcard copy won't have the right name (`git`) but the directory may shadow the explicit rule depending on cfgcaddy's glob precedence. Also, the `ignore:` section lists `*stapler*` as a glob pattern, which would match `stapler-scripts/git-ssh-fallback` and suppress the link entirely. Verify whether the ignore pattern applies before or after explicit link rules. **Remediation**: audit cfgcaddy's ignore-vs-explicit-link precedence rules; confirm `*stapler*` does not suppress the explicit link entry. If ignore applies to explicit rules, rename the script (e.g., `wrapper-scripts/git-ssh-fallback`) or add an exception. Integration-test `cfgcaddy link --dry-run` before deploying.

- [ ] **Tasks 2.1.1a (AuthCache naming)** — ILLEGAL STATE: The plan states the cache file name is `${host}.auth` (e.g., `github.com.auth`). A hostname containing only alphanumerics and dots is generally safe, but self-hosted or enterprise hostnames may include hyphens, underscores, or port suffixes (e.g., `git.corp.example.com:8443`). A port suffix introduces `:` which is illegal in macOS/Linux filenames. The domain glossary defines `RemoteHost` as "no port" but the extraction logic in Task 1.3.1a extracts host via `${url#https://}` then `${host%%/*}` — this strips the path but does NOT strip a port (e.g., `https://git.corp.com:8443/org/repo` → `git.corp.com:8443`). The colon would either cause a subtle `touch` failure or create a file with a literal colon in its name (valid on Linux, silently broken on macOS HFS+ paths). **Remediation**: In `_extract_host()`, also strip the port: after `${host%%/*}`, apply `${host%%:*}`. Add this to the RemoteHost newtype definition and test explicitly with a port-bearing URL.

- [ ] **Task 1.2.1a (NetworkSubcommand passthrough sets env var)** — SIDE EFFECT POLLUTION: The plan specifies `export GIT_SSH_FALLBACK_ACTIVE=1; exec "$REAL_GIT" "$@"` on the non-network passthrough path. Setting `GIT_SSH_FALLBACK_ACTIVE` even for pure local commands (e.g., `git status`) means every subprocess spawned from that git process will see the guard variable, which is harmless but wastes an env slot and leaks a debug signal unnecessarily. More importantly, the recursion guard at the top is checked before the passthrough: if the RecursionGuard is set, the script execs immediately — so the local-passthrough path is never reached when called from within a git hook fired by a previous wrapper invocation. The logic is correct but the export on local passthrough is unnecessary noise. **Remediation** (low severity within this concern): Only export `GIT_SSH_FALLBACK_ACTIVE=1` immediately before `exec`ing `RealGit` in the SSH-fallback path. For local passthroughs that exec RealGit with the original `$@`, the guard is not needed because RealGit won't call back into the wrapper. This marginally reduces env pollution and makes the guard's purpose clearer.

- [ ] **Epic 2.2 / Task 2.2.1a (SSH agent detection: Priority 6 subshell glob)** — PARSE ROBUSTNESS: Priority 6 uses `$(ls -t "$HOME/.ssh/agent"/s.*.agent.* 2>/dev/null | head -1)`. Two issues: (a) `ls -t` + `head -1` gets the most recent socket but the path is passed to `[ -S ... ]` as a shell word — if the path contains spaces (unlikely for socket paths but theoretically possible), the test will fail silently. (b) `ls` output parsing is generally fragile. The plan uses `set -euf` globally — the `2>/dev/null` suppresses the error but `ls` returning empty is fine; however, if the glob expands to zero matches, `ls` exits non-zero with stderr suppressed, and `$()` captures empty string. With `set -u`, reading an empty assignment would not be an issue since the variable is only set if non-empty. This is likely correct as written, but worth an explicit comment. **Remediation**: Replace with `find "$HOME/.ssh/agent" -name 's.*.agent.*' -maxdepth 1 2>/dev/null | sort -r | head -1` for portability; or add an inline comment explaining why `ls -t` is acceptable here.

- [ ] **Epic 2.3 / Task 2.3.1a (ConfigInjection assumes GIT_CONFIG_COUNT is unset)** — ILLEGAL STATE: The plan sets `GIT_CONFIG_COUNT=1` unconditionally. If the calling environment already has `GIT_CONFIG_COUNT` set (e.g., a parent process or IDE that injects its own git config overrides), hardcoding `GIT_CONFIG_COUNT=1` would silently overwrite the existing count, discarding all parent-injected config keys and potentially breaking git behavior for that invocation. The plan mentions "Reject approach if GIT_CONFIG_COUNT would overflow existing env (defensive)" in the pattern table but Task 2.3.1a does not implement this check. **Remediation**: In `_inject_ssh_config()`, read the current value of `GIT_CONFIG_COUNT` (default 0 if unset), add 1 to compute the new count, set `GIT_CONFIG_KEY_<N>` and `GIT_CONFIG_VALUE_<N>` at the existing count's index, and set `GIT_CONFIG_COUNT=$((existing + 1))`. This composes correctly with callers that already set config overrides.

- [ ] **Story 3.2.1 / Task 3.2.1a (self-test tests implementation details, not contracts)** — TESTABILITY: The self-test checks `_find_real_git` returns a path and `_extract_subcommand` / `_is_network_op` return expected values. These test internal function contracts, not the observable behavior of the wrapper. For a PATH-intercepting script, the meaningful contract test is: given specific env + args, what exit code and stderr does the wrapper produce? A self-test that only exercises internal helpers would not catch a regression in the main control flow (e.g., wrong branching after `_is_network_op`). **Remediation**: Augment `--self-test` with at least one end-to-end scenario: a mock that stubs `gh` with an exit-code-controlled script, verifying that `GIT_CONFIG_COUNT=1` appears in the exec'd environment. This can be done portably in POSIX sh using a temp-dir mock `gh` script. Alternatively, document that `--self-test` is a smoke test only and that full behavioral testing requires the smoke test suite in Story 3.2.1.

## Nitpicks

- `set -euf` at the top of the script combined with `_detect_ssh_agent()` returning 1 on failure requires care: in POSIX sh with `set -e`, a function returning non-zero in a context that is not explicitly guarded (e.g., `if _detect_ssh_agent; then`) will abort the script. The plan correctly uses `if`-guarded invocations throughout, but implementers should note this and add a comment.

- The glossary defines `AuthCache` file as `<host-hash>.auth` but the implementation (Task 2.1.1a) uses `${host}.auth` directly (no hash). The glossary is inconsistent with the implementation plan. Minor documentation debt — update the glossary to say `<host>.auth` or document why hashing was dropped (it was: "alphanumeric hostname is safe as filename; no hash needed").

- `GIT_SSH_COMMAND` sets `StrictHostKeyChecking=accept-new` (Task 2.3.1b). This silently accepts any new host key. For a personal dotfiles tool this is an acceptable tradeoff, but it is a security decision that should be called out in comments in the script, not just in the plan. The requirements doc acknowledges this tradeoff but the code will not have this context without an inline comment.

- The `--self-test` flag (Task 3.2.1a) intercepts before git logic and does not exec RealGit. If a user accidentally deploys the wrapper as `git` and runs `git --self-test`, git itself would never receive the flag. This is fine for a personal tool but should be documented in the script header.

- The requirements observability section says the fallback log line format is `[git-ssh-fallback] gh auth not found for HOST, falling back to SSH` but the plan's `FallbackWarning` domain glossary and Task 2.3.1c specify `warning: falling back to SSH (gh not authenticated for HOST)` (no prefix, different phrasing). These are inconsistent. Pick one and document it as the canonical format.
