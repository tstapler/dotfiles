# Pre-mortem: git-ssh-fallback
Date: 2026-06-24

## Failure Modes

1 | **gh Keychain timeout causes false "unauthenticated" on a hot machine** — the 3-second Keychain timeout (cli/cli#13317) causes `gh auth token` to return non-zero even when the user is authenticated, permanently routing all HTTPS operations through SSH fallback; the user never notices until they hit a repo where their SSH key is not configured, at which point the error message is "Permission denied (publickey)" with no indication the wrapper is involved. | Symptom: `warning: falling back to SSH (gh not authenticated for github.com)` appears on every `git fetch` even though the user ran `gh auth login` last week. | Prevention: add a validation test to `validation.md` that exercises the auth-check path while simulating a Keychain timeout (mock `gh` to exit 124) and confirms the wrapper treats timeout differently from a genuine unauthenticated state — e.g., on timeout, log a distinct `warning: gh auth check timed out, assuming authenticated` and skip the fallback rather than triggering it; amend Task 2.1.1b to handle exit 124 separately from exit 1. | P1

2 | **IDE (VSCode/JetBrains) bypasses the PATH wrapper entirely** — IDEs resolve git via `git.path` / Preferences → Version Control hardcoded to `/usr/bin/git`, so the wrapper is never invoked for IDE-driven fetch/push; the user experiences HTTPS auth failures inside the IDE while the terminal appears to work fine, making the problem appear non-deterministic. | Symptom: `git fetch` from the terminal succeeds silently via SSH fallback, but VSCode source control pane shows a persistent authentication error badge and fails to sync. | Prevention: add an explicit step in Epic 3.1 (deployment) to verify IDE coverage — at minimum document that `git.path` in VSCode and the JetBrains Git executable setting must be pointed at `~/.local/bin/git`; better still, add a `--self-test` assertion that checks `which git` resolves to the wrapper and emits a warning if not. | P2

3 | **The AuthCache causes stale "authenticated" for hours after a token expires** — the 5-minute TTL cache writes an auth file on first success; if the gh token expires or is revoked mid-session the wrapper keeps returning "authenticated" from cache and skips the SSH fallback; git then fails with a 401 from the remote, but the error surface is git's own HTTPS error — no wrapper diagnostic, no hint that the auth cache is stale. | Symptom: `git push` fails with `remote: Invalid username or password` immediately after a `gh auth logout` or token rotation, but `GIT_SSH_FALLBACK_DEBUG=1` shows `auth check: cached (authenticated)` — the wrapper did not trigger fallback. | Prevention: add an acceptance criterion in Story 2.1.1 that when a cached-authenticated operation fails (git exits non-zero), the wrapper invalidates the AuthCache file and optionally retries via SSH; alternatively document the cache as a latency optimization only and explicitly set the TTL to 0 or 1 minute to limit the blast radius of stale state. | P2

4 | **Recursive clone with submodules hangs silently in headless contexts** — `git clone --recurse-submodules` spawns a separate git process per submodule, each of which invokes the wrapper; if the submodule remote host is new (not in `~/.ssh/known_hosts`) the SSH fallback's `StrictHostKeyChecking=accept-new` should handle it, but if the SSH agent is a 1Password agent that requires the desktop app and the session is headless (Claude Code subagent, cron), `_detect_ssh_agent()` either returns stale-socket false-positive and then the actual SSH connection hangs waiting for the agent, or the parallel submodule clones hit the 3-second `ssh-add -l` liveness timeout serially, adding `N * 3s` overhead. | Symptom: `git clone --recurse-submodules https://github.com/org/large-monorepo` hangs for 60+ seconds without output in a Claude Code subagent session, then fails with a timeout. | Prevention: add a submodule regression test in Story 3.2.1 that clones a repo with at least 3 submodules in a simulated headless env (SSH_AUTH_SOCK unset); verify total time stays under 10 seconds and no hang occurs; the liveness probe timeout in Task 2.2.1a should be reduced from 3s to 1s for the submodule case, or the timeout should be surfaced to the user immediately rather than retried per submodule. | P1

5 | **The non-gh-managed host path (Epic 2.4) always rewrites to SSH when any SSH agent is present, breaking repos that require HTTPS tokens for non-GitHub forges** — the logic in Task 2.4.1b proactively rewitres non-gh-managed HTTPS to SSH whenever a live SSH agent is found, regardless of whether the user actually has SSH keys provisioned for that host; for hosts like internal Bitbucket or Gitea that use HTTPS PATs stored in the OS Keychain (not gh), the SSH rewrite breaks auth silently — the SSH connection fails with "Permission denied (publickey)" rather than succeeding with the HTTPS credential. | Symptom: `git fetch` against `git.corp.example.com` starts failing after dotfiles deploy; the user does have HTTPS credentials configured but no SSH key for that host; `warning: falling back to SSH (gh not authenticated for git.corp.example.com)` appears even though HTTPS would have worked. | Prevention: change the non-gh-managed host logic from "SSH if agent present" to "SSH only if SSH key for host is in known_hosts OR ssh-add -l lists a key that matches the host"; add an acceptance criterion to Story 2.4.1 that explicitly tests a non-gh host where HTTPS works but SSH would not, and confirms the wrapper does not rewrite. | P1

---

## P1 Items (address before implementation)

- [x] Failure #1 (gh Keychain timeout → false unauthenticated) — **RESOLVED**: Task 2.1.1b updated to treat exit 124 as "assume authenticated, skip fallback"; validation plan covers this exit code path.
- [x] Failure #4 (submodule recursive clone hangs headless) — **RESOLVED**: Task 2.2.1a `ssh-add -l` timeout reduced from 3s to 1s; headless+submodule scenario noted in Known Limitations of plan.md.
- [x] Failure #5 (non-gh host proactive SSH rewrite breaks HTTPS-only forges) — **RESOLVED**: Task 2.4.1b updated with `known_hosts` gate (`ssh-keygen -F "$host"`) as required positive signal before SSH rewriting.
