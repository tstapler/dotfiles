# UX Design: mcp-context-filter

**Date**: 2026-07-02
**Phase**: Design (Phase 3 — pre-implementation)
**Authored by**: UX design pass, fed from requirements.md + research/ux.md

---

## Overview

Seven user-facing surfaces are designed in this document. Each surface has an ASCII mockup, an interaction flow, and error edge-case handling. UX acceptance criteria follow.

### Surfaces

| # | Surface | Primary user action |
|---|---------|---------------------|
| 1 | `mcp-proxy.toml` config file | Author / prune allowlists |
| 2 | First-run / bootstrap (`mcp-proxy init` → `analyze`) | Discover and configure proxy for first time |
| 3 | `mcp-proxy sync` command | Detect and resolve schema drift |
| 4 | Session-start log output | Verify token savings at session start |
| 5 | Error states | Recover from failures |
| 6 | Claude Code config helper (`mcp-proxy install`) | Wire proxy into Claude Code `.mcp.json` |
| 7 | `mcp-proxy serve` (stdio subprocess) | Not user-driven — spawned by Claude Code as MCP transport |

## Surface 1: mcp-proxy.toml Config File

### Purpose

The config file is the user's primary control surface. It must be readable without documentation, writable without memorizing syntax, and scannable when looking for a specific server block.

### ASCII Mockup — Full Generated File

```toml
# mcp-proxy.toml
# Generated by: mcp-proxy init  2026-07-02T09:00:01Z
# !! Restart proxy to apply changes (hot-reload not supported) !!
#
# Config last applied: 2026-07-02T09:00:01Z
# Managed by: claude-proxy-rs  v0.4.0

# ─── Global Settings ──────────────────────────────────────────────────────────

[global]
enabled           = true
compression_level = "light"   # off | light | aggressive
log_token_savings = true
upstream_timeout_ms = 5000

# ─── Server Allowlists ────────────────────────────────────────────────────────
# Each block controls which tools one upstream server exposes to Claude.
# Token estimates are approximations based on average schema sizes.
# Run 'mcp-proxy sync' to refresh tool counts and detect upstream changes.

[servers.atlassian]
# Before: ~13,000 tokens (42 tools)  →  After: ~3,200 tokens (11 tools)  [75% saved]
# Last scanned: 2026-07-02 | 31 tools blocked
upstream_url = "https://mcp.atlassian.com/mcp"
allow = [
    "addCommentToJiraIssue",
    "createConfluencePage",
    "createJiraIssue",
    "editJiraIssue",
    "getAccessibleAtlassianResources",
    "getConfluencePage",
    "getJiraIssue",
    "search",
    "searchConfluenceUsingCql",
    "searchJiraIssuesUsingJql",
    "updateConfluencePage",
]

[servers.brave-search]
# Before: ~1,200 tokens (2 tools)  →  After: ~1,200 tokens (2 tools)  [0% saved]
# Last scanned: 2026-07-02 | 0 tools blocked  (small server — no filtering needed)
upstream_url = "https://api.search.brave.com/mcp"
allow = [
    "brave_local_search",
    "brave_web_search",
]

[servers.slack]
# Before: ~9,500 tokens (38 tools)  →  After: ~1,800 tokens (8 tools)  [81% saved]
# Last scanned: 2026-07-02 | 30 tools blocked
# NEW: slack_huddle_start  — added to upstream since last scan; add below to enable
upstream_url = "https://mcp.slack.com/mcp"
allow = [
    "slack_read_channel",
    "slack_read_thread",
    "slack_read_user_profile",
    "slack_search_channels",
    "slack_search_public",
    "slack_search_users",
    "slack_send_message",
    "slack_send_message_draft",
]
```

### Config Annotation Conventions

| Annotation | Placement | Meaning |
|------------|-----------|---------|
| `# NEW: <tool>` | After server header | Tool added upstream since last scan; not yet in allowlist |
| `# REMOVED: <tool>` | After server header | Tool in allowlist no longer found upstream (staleness warning) |
| `# Last scanned: <date>` | Per-server comment | When `mcp-proxy sync` last ran for this server |
| `# Before: / After: / saved` | Per-server comment | Token cost estimate; updated by `mcp-proxy sync` |
| `!! Restart proxy to apply changes !!` | File header | Always present; prevents the "why didn't my edit work?" confusion |

### Interaction Flow — Editing the Allowlist

```
User goal: Remove slack_send_message_draft from Slack allowlist

Step 1  Open mcp-proxy.toml in editor
Step 2  Search for "[servers.slack]" — found in <1s (alphabetical order, one block per server)
Step 3  Delete the "slack_send_message_draft" line from the allow array
Step 4  Save file
Step 5  Restart claude-proxy-rs (launchctl kickstart or kill + relaunch)
Step 6  Start new Claude Code session
Step 7  Check session-start log — Slack now shows 7 tools instead of 8

Total steps: 7 (5 user actions, 2 confirmations)
```

### Error Edge Cases — Config File

**Typo in field name**

```
# User writes: alowlist = [...]  instead of allow = [...]

mcp-proxy: config parse error
  file: /Users/tyler/.claude/mcp-proxy.toml  line 34
  error: unknown field `alowlist` in table `servers.slack`
  hint:  did you mean `allow`?
  fix:   run `mcp-proxy validate` to check the full config before restarting
```

**Duplicate server block**

```
mcp-proxy: config parse error
  file: /Users/tyler/.claude/mcp-proxy.toml  line 47
  error: duplicate key `slack` in table `servers`
  hint:  each server must have one block; merge the duplicate entries
```

**Tool name with unsupported characters**

```
mcp-proxy: config validation warning
  server: slack  tool: "slack read channel"   (contains space)
  hint:  tool names are case-sensitive and must match upstream exactly
         upstream name is: slack_read_channel
  action: no change applied; correct the name and restart
```

**global.enabled = false (passthrough mode warning)**

```
mcp-proxy: running in PASSTHROUGH mode  (global.enabled = false)
  all tools exposed to Claude — no filtering active
  token savings: 0 tokens saved this session
  to enable filtering: set enabled = true in mcp-proxy.toml and restart
```

## Surface 2: First-Run / Bootstrap Experience

### Purpose

A user who has never set up the proxy must reach "filtering active, tokens saved" with minimal friction and zero risk of breaking their currently-used tools. The bootstrap is a 2-step flow: `init` (day 0) then `analyze` (day 7).

**Supported flags**: `--yes` (skip [Y/n] confirmation, apply allowlist without interaction — for scripting), `--plain` (ASCII-only output), `--json` (emit JSON to stdout for scripting), `--quiet` (suppress all output except errors and final result line)

### Error Edge Cases — Bootstrap

**settings.json not found**

```
mcp-proxy init: cannot find Claude Code settings
  searched: ~/.claude/settings.json
  error:    file not found
  hint:     is Claude Code installed? expected path: ~/.claude/settings.json
            if installed in a non-standard location, set:
            CLAUDE_SETTINGS_PATH=/path/to/settings.json mcp-proxy init
```

**No MCP servers registered**

```
mcp-proxy init: found 0 registered MCP servers in settings.json
  hint: MCP servers are configured in the mcpServers section of settings.json
        if servers are denied (deniedMcpServers), they will not appear
        run 'mcp-proxy install' first to ensure servers are registered
```

**Observation log empty at analyze time (zero calls recorded)**

```
mcp-proxy analyze: no tool calls recorded in mcp-proxy-calls.log
  observation period: 2026-06-25 → 2026-07-02 (7 days)
  hint: this can happen if:
    1. The proxy was not running during this period (check launchctl status)
    2. No Claude Code sessions were started
    3. The log file was deleted or rotated

  Options:
    a) Extend observation: mcp-proxy analyze --days 14
    b) Generate a permissive allowlist (all tools) and prune manually:
       mcp-proxy analyze --all-tools
    c) Abort: no config changes made
  Choice [a/b/c]: _
```

**User runs analyze before 7 days**

```
mcp-proxy analyze  (3-day observation: 2026-06-29 → 2026-07-02)
  WARNING: Only 3 days of data collected (7 recommended)
  Tool call patterns may be incomplete — infrequently-used tools may not appear.

  Continue anyway? [y/N]: _
```

## Surface 3: `mcp-proxy sync` Command

### Purpose

Detects schema drift: tools added or removed from upstream servers since the last scan. This is the mitigation for the "tool silently disappears" failure mode identified in research/ux.md.

**Supported flags**: `--yes` (accept all recommended changes without interactive prompts — for scripting), `--plain` (ASCII-only output), `--json` (emit JSON to stdout for scripting), `--quiet` (suppress all output except errors and final result line)

### Interaction Flow — Sync on Schema Change

```
Trigger: User notices "slack_create_canvas not working" or runs sync proactively

Step 1   Run: mcp-proxy sync
Step 2   Read drift report — see NEW / STALE / REMOVED for each server
Step 3   Choose action per server (add / ignore / review)

```

### Error Edge Cases — Sync

**All servers unreachable (network down)**

```
mcp-proxy sync: all 14 servers unreachable
  error: no upstream servers responded within 5000ms
  hint:  check network connectivity: ping mcp.slack.com
         if behind VPN, ensure VPN is connected
  last successful sync: 2026-07-01T09:00:00Z (1 day ago)
  no changes made to mcp-proxy.toml
```

**Partial server failure (some reachable, some not)**

```
Sync completed with warnings:
  11 servers: ok
   3 servers: unreachable (figma, google-drive, zoom)

  Unreachable servers are shown as "unverified" — their last-known state
  is preserved in mcp-proxy.toml comments.
  No changes applied to unreachable servers.
```

**Sync run while proxy is actively handling requests**

```
mcp-proxy sync: proxy is currently running
  hint: sync reads upstream servers directly and writes mcp-proxy.toml
        the running proxy will use the OLD config until restarted
        restart after sync to apply changes:
          launchctl kickstart -k gui/$(id -u)/com.tyler.claude-proxy-rs
```

## Surface 4: Session-Start Log Output

### Purpose

Each time a new Claude Code session starts, the proxy logs a one-line summary per server showing the filtering result. This is the user's feedback that "the proxy is working." The success path must be readable at a glance; no output is needed on the happy path once the user trusts the proxy.

### ASCII Mockup — INFO-Level Session Start Log

```
2026-07-02T14:00:01Z INFO  mcp_filter: session started
2026-07-02T14:00:01Z INFO  mcp_filter: atlassian   tools=11/42  tokens=3,200  (saved 9,800  75%)
2026-07-02T14:00:01Z INFO  mcp_filter: brave-search tools= 2/ 2  tokens=1,200  (saved     0   0%)
2026-07-02T14:00:01Z INFO  mcp_filter: slack       tools= 8/38  tokens=1,800  (saved 7,700  81%)
2026-07-02T14:00:01Z INFO  mcp_filter: [11 more servers...]      tokens=3,200  (saved 7,600  70%)
2026-07-02T14:00:01Z INFO  mcp_filter: session total             tokens=9,400  (saved 33,200 78%)
                                                                  context used: 4.7% of 200k
```

### ASCII Mockup — Passthrough Mode Warning

```
2026-07-02T14:00:01Z WARN  mcp_filter: running in PASSTHROUGH mode
                           global.enabled = false in mcp-proxy.toml
                           tokens: 42,600 (no filtering active — 21.3% of context window)
                           to enable filtering: set enabled = true and restart proxy
```


### ASCII Mockup — Stale Allowlist Warning at Session Start


### Interaction Flow — Verifying a Working Session

```
User goal: Confirm proxy reduced token load in a new session

Step 1   Open session log:  tail -n 20 /tmp/claude-proxy-rs.app.log
Step 2   Find "mcp_filter: session started" lines
Step 3   Read "session total" line — tokens=9,400  context used=4.7%
Step 4   Confirm: filtering is active, 78% savings achieved

Total steps: 4 (read-only verification)
No action required on success path.
```

### Log Level Behavior Summary

| Event | Level | Output |
|-------|-------|--------|
| Session start, filtering active | INFO | One line per server + total |
| Session start, passthrough mode | WARN | Warning + token cost + fix hint |
| Session start, unconfigured servers present | WARN | List of unconfigured servers + hint |
| Session start, stale allowlist tool | WARN | Tool name + sync hint |
| Tool call at DEBUG level | DEBUG | Per-call trace (suppressed at INFO) |
| Tool call at INFO level | (silent) | No per-call noise at INFO |
| Proxy healthy, nothing changed | (silent) | No output |

## Surface 5: Error States

### Purpose

Error states must have no dead ends. Every error message must name the problem, identify the cause, and offer a concrete next action. Claude's error view (MCP JSON-RPC) and the user's log view are designed separately per research/ux.md.

### 5a — Upstream Server Unreachable

**What Claude sees (MCP tools/list response):**

```json
{
  "jsonrpc": "2.0",
  "id": 1,
  "error": {
    "code": -32603,
    "message": "upstream server unreachable: slack (timeout after 5000ms)",
    "data": {
      "server": "slack",
      "upstream_url": "https://mcp.slack.com/mcp",
      "consecutive_failures": 3,
      "hint": "the Slack MCP server appears to be unavailable; try again or inform the user"
    }
  }
}
```

**What Claude sees (MCP tools/call response for affected tool):**

```json
{
  "jsonrpc": "2.0",
  "id": 42,
  "error": {
    "code": -32603,
    "message": "tool unavailable: slack_send_message (upstream server slack is unreachable)",
    "data": {
      "tool": "slack_send_message",
      "server": "slack",
      "hint": "the Slack MCP server is currently unavailable"
    }
  }
}
```

**What the user sees in log:**

```
2026-07-02T14:32:01Z WARN  mcp_proxy: upstream unreachable
                           server=slack  url=https://mcp.slack.com/mcp
                           timeout_ms=5000  consecutive_failures=3
                           last_success=2026-07-02T14:27:43Z  (4 minutes ago)
2026-07-02T14:32:01Z ERROR mcp_proxy: fail-fast triggered for server
                           server=slack
                           message="failing fast after 3 consecutive timeouts"
                           hint="diagnose: curl -s https://mcp.slack.com/mcp/health"
```

**Recovery path:**

```
Step 1   Identify: log says "upstream unreachable: slack"
Step 2   Diagnose: curl -s https://mcp.slack.com/mcp/health
Step 3a  If network issue: fix connectivity, restart Claude Code session
Step 3b  If Slack outage: wait, retry later (no proxy action needed)
Step 3c  If auth expired: re-authenticate via Claude Code's /auth command
```

### 5b — Authentication Failure

**What Claude sees (MCP tools/list or tools/call):**

```json
{
  "jsonrpc": "2.0",
  "id": 5,
  "error": {
    "code": -32603,
    "message": "authentication failed: atlassian (HTTP 401 from upstream)",
    "data": {
      "server": "atlassian",
      "upstream_status": 401,
      "hint": "re-authenticate: restart Claude Code and re-authorize the Atlassian connector"
    }
  }
}
```

**What the user sees in log:**

```
2026-07-02T10:15:00Z WARN  mcp_proxy: auth failure
                           server=atlassian  status=401
                           hint="session token may have expired"
                           action="restart Claude Code to trigger re-authentication"
```

| Error | Proxy starts? | Claude sees | Log level | Recovery action |

## Terminal Compatibility Requirements

All CLI output must degrade gracefully when:
1. NO_COLOR env var is set (https://no-color.org/) — all ANSI color codes suppressed; INFO/WARN/ERROR markers retained as plain text
2. `--plain` flag or `TERM=dumb` — replace all Unicode box-drawing characters with ASCII equivalents:
   - `═` → `=`, `─` → `-`, `┌┐└┘├┤│` → `+` and `|`
   - Token tables use `+---+---+` ASCII borders
3. `--json` flag for any command that produces structured output (init, analyze, sync, status) — emit JSON to stdout for scripting; human text to stderr
4. `--quiet` flag — suppress all output except errors and the final result line

```
┌────────────┐ → +------------+

├────────────┤ → +------------+
```

## Surface 7: `mcp-proxy serve` (stdio subprocess)

### Purpose

`mcp-proxy serve --server <name>` is the core runtime command. It is spawned automatically by Claude Code as a stdio subprocess at session start — the user never runs it directly. Its stdout and stdin are consumed exclusively by Claude Code's MCP JSON-RPC transport. Its design constraints differ from every other surface: it has no interactive user-visible output on the happy path, and all human-visible output goes to stderr.

### Execution Model

- **Spawned by**: Claude Code at session start, as a `mcpServers` stdio entry in `settings.json`
- **Not run directly by the user** — invoked via config, not the terminal
- **stdout/stdin**: Consumed entirely by Claude Code's MCP transport; contains only valid JSON-RPC messages; no human-readable text may ever appear on stdout
- **stderr**: Human-visible log output; the session-start token log (Surface 4) goes to stderr and is captured by the log file

### ASCII Mockup — Normal Operation (stderr)

```
# Written to stderr (captured in log file); never to stdout

2026-07-02T14:00:01Z INFO  mcp_proxy: serving atlassian allowlist=11/42 tools
```

One line per session start per subprocess. Silent thereafter on the happy path.

### ASCII Mockup — Config Error (stderr + exit)

```
# stderr output when config file is missing:

2026-07-02T14:00:01Z ERROR mcp_proxy: config file not found
                           path=/Users/tyler/.claude/mcp-proxy.toml
                           hint="run mcp-proxy init to generate a config, then restart Claude Code"
mcp_proxy: fatal — exiting with code 1
```

### Error States

| Error condition | Behavior | Exit code | Output |
|-----------------|----------|-----------|--------|
| Config file not found | Write ERROR to stderr + exit | 1 | Path + `mcp-proxy init` hint |
| Config parse error | Write ERROR to stderr + exit | 1 | Line number + did-you-mean hint |
| Upstream unreachable at startup | Write WARN to stderr, continue serving | (no exit) | Server name + URL + `curl` diagnostic hint |
| tokio runtime panic / unexpected crash | Terminate | 2 | Panic trace to stderr |
| Clean session end (stdin EOF from Claude Code) | Exit cleanly | 0 | None |

### Interaction Flow

`mcp-proxy serve` has no interactive flow — it is not user-driven. The only user-visible action is inspecting the log file after an unexpected failure:

```
Trigger: Claude Code session starts; no tools appear for a server

Step 1   Check log: tail -n 30 /tmp/claude-proxy-rs.app.log
Step 2   Look for ERROR lines from mcp_proxy at session-start timestamp
Step 3a  Config error (exit 1): run mcp-proxy validate, fix errors, restart
Step 3b  Upstream unreachable (WARN, no exit): run curl -s <upstream-url>, diagnose connectivity
Step 3c  Unexpected crash (exit 2): inspect panic trace, report issue

Total steps: 3 (log-read only; no interactive prompts)
```

### Design Constraints Summary

| Constraint | Rule |
|------------|------|
| C7.1 | stdout contains only valid JSON-RPC; no human-readable text ever written to stdout |
| C7.2 | Config error (missing file or parse error) exits with code 1 and writes diagnostic message to stderr within 100ms of startup |
| C7.3 | Upstream unreachable at startup does NOT cause exit; session continues without that server's tools; WARN written to stderr |
| C7.4 | Startup acknowledgment writes exactly one line to stderr per session: `mcp_proxy: serving <server_name> allowlist=<n>/<total> tools` |

**Note**: Surface 7 has 1 mockup (stderr format), 0 interactive flows (not user-driven), and 4 error cases.
