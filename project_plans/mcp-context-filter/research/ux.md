## 1. Config File UX

### The Verbosity Spectrum in Transparent Proxy Configs

Developer infrastructure config files fall on a spectrum from minimal (only what differs from defaults) to exhaustive (every field explicit). The right choice is shaped by how often the file is edited and what it generates.

**Minimal (gitconfig, ssh config)**
```toml
[core]
    autocrlf = false
```
Write once, rarely revisit. Works because the mental model is stable.

**Explicit-but-commented (rsync daemon, nginx)**
Every non-default has a comment explaining the tradeoff. Verbosity justified because the file is the primary documentation.

**Generated-then-owned (renovate.json, prettier, .prettierrc)**
Tool writes the first version; user extends. Works when the generator outputs well-chosen defaults and leaves human-readable comments inline.

### What mcp-proxy.toml Should Look Like

The config has two distinct concerns that should be visually separated: **global proxy settings** and **per-server allowlists**. Users will scan for the server they care about — make that scannable.

**Recommended structure:**

```toml
# mcp-proxy.toml
# Generated by: claude-proxy-rs mcp-proxy init (2026-07-01)
# Edit allowlists below. Global settings rarely need changes.

[global]
enabled = true
compression_level = "light"   # off | light | aggressive
log_token_savings = true
upstream_timeout_ms = 5000

# ─── Server Allowlists ────────────────────────────────────────────────────────

[servers.slack]
# 9,500 tokens → estimated 1,800 tokens after filtering
# Last scanned: 2026-07-01 | Tools available: 38 | Tools allowed: 8
upstream_url = "https://mcp.slack.com/mcp"
allow = [
    "slack_read_channel",
    "slack_send_message",
    "slack_search_public",
    "slack_search_users",
    "slack_read_thread",
    "slack_read_user_profile",
    "slack_search_channels",
    "slack_send_message_draft",
]

[servers.atlassian]
# 13,000 tokens → estimated 3,200 tokens after filtering
# Last scanned: 2026-07-01 | Tools available: 42 | Tools allowed: 11
upstream_url = "https://mcp.atlassian.com/mcp"
allow = [
    "getJiraIssue",
    "searchJiraIssuesUsingJql",
    "editJiraIssue",
    "addCommentToJiraIssue",
    "createJiraIssue",
    "getConfluencePage",
    "searchConfluenceUsingCql",
    "search",
    "getAccessibleAtlassianResources",
    "createConfluencePage",
    "updateConfluencePage",
]
```

**Design decisions:**

1. **Comment the token math on each server block.** This is the whole point of the tool. Making the savings visible at the config level reinforces the mental model ("I chose these 8 tools; that cut 83% of Slack's schema tokens").

2. **One block per server, alphabetical.** Users know the server name, not the index. Alphabetical ordering makes `[servers.slack]` findable in 1s.

3. **`allow =` not `deny =`.** Allowlist semantics are safer (unknown tools fail closed) and the list is shorter (8 vs 30). The UX of writing a positive list is also faster: "what do I use?" vs "what don't I want?".

4. **"Last scanned" comment.** Tells the user whether this list reflects recent usage or was guessed. If the proxy adds a new tool it can append `# NEW: <tool_name>` for tools added to the upstream server since the last scan.

5. **Never require quoting tool names.** TOML bare strings don't work here (hyphens, underscores, slashes are common). Use `allow = ["quoted_name"]` consistently — the array format is unambiguous.

### Auto-Generate vs Manual

The first allowlist should be auto-generated. The bootstrapping command should output a file the user can immediately use, with conservative defaults (all tools from the last 30 days plus a buffer), not a minimal list that breaks immediately.

The user then prunes. This is the right model: generated config is a ceiling, not a floor.

```
# init output format:
[servers.slack] # 38 tools → generated 8-tool allowlist from 30-day scan
# Tools kept: called ≥1 time in past 30 days
# Tools dropped: 30 never-called tools (29,400 estimated tokens saved)
```

Do NOT require the user to manually look up upstream URLs. The init command should interrogate Claude Code's config to find registered MCP servers and write the URLs automatically.

## 2. Error UX

### Two Error Audiences, Two Surfaces

The MCP context filter has two error audiences:

- **Claude** — receives the result of a `tools/call` or `tools/list` request. Claude's error experience is defined by what the proxy returns in MCP JSON-RPC format.
- **Tyler** — reads log files, sees proxy startup failures, notices broken behavior. Tyler's error experience is defined by structured log lines in `/tmp/claude-proxy-rs.app.log`.

These should never be conflated. Log verbosity does not affect what Claude sees.

### When Upstream Server Is Down

**What Claude should see:**

MCP `tools/list` call — server unavailable path:
```json
{
  "jsonrpc": "2.0",
  "id": 1,
  "error": {
    "code": -32603,
    "message": "upstream server unreachable: slack (timeout after 5000ms)",
    "data": {"server": "slack", "upstream_url": "https://mcp.slack.com/mcp"}
  }
}
```

For `tools/call` when the specific tool's server is down:
```json
{
  "jsonrpc": "2.0",
  "id": 42,
  "error": {
    "code": -32603,
    "message": "tool unavailable: slack_send_message (upstream server slack is unreachable)"
  }
}
```

The error message Claude sees should be **actionable within the session context** — Claude can tell the user "the Slack MCP server appears to be unavailable" rather than failing silently or showing a generic proxy error.

**What Tyler should see in logs:**

```
2026-07-02T14:32:01Z WARN mcp_proxy: upstream unreachable
  server=slack url=https://mcp.slack.com/mcp timeout_ms=5000
  consecutive_failures=3 last_success=2026-07-02T14:27:43Z
2026-07-02T14:32:01Z ERROR mcp_proxy: fail-fast triggered for server
  server=slack message="failing fast after 3 consecutive timeouts"
  hint="run: curl -s https://mcp.slack.com/mcp/health to diagnose"
```

The hint in the log line is the equivalent of the rustc `--explain` pattern — it gives a concrete diagnostic command without requiring the user to know the proxy internals.

### When a Tool Is Filtered (Not In Allowlist)

This is the subtler case. If Claude attempts to call a tool that has been filtered out, the proxy has two options:

1. **Return it in `tools/list`**: not an option — the whole point is not to expose it.
2. **Claude never calls it**: ideal steady state — Claude doesn't know the tool exists, so it never tries.
3. **Claude somehow calls it anyway** (e.g., from a system prompt stub or Phase 3 dynamic registration): the proxy should return a clean `tool_not_found` error, not a mysterious timeout.

The `tool_not_found` path is the critical edge case to design explicitly:

```json
{
  "jsonrpc": "2.0",
  "id": 15,
  "error": {
    "code": -32601,
    "message": "tool not found: slack_create_canvas (tool exists but is not in allowlist)",
    "data": {"tool": "slack_create_canvas", "server": "slack", "allowlist_hint": "add to mcp-proxy.toml to enable"}
  }
}
```

The `allowlist_hint` surfaces the remediation in the error, which Claude can relay to the user: "The tool exists but is filtered — you can add it to your allowlist."

### Startup / Config Parse Errors

These happen at LaunchAgent startup, before any MCP requests. They should:
1. Be fatal (don't start with broken config)
2. Print a human-readable error to stderr that ends up in the `.error.log` file
3. Include the file path and line number when possible

```
2026-07-02T09:00:01Z ERROR config: failed to parse mcp-proxy.toml
  path=/Users/tyler/.claude/mcp-proxy.toml line=23
  error="unknown field `alowlist` (did you mean `allow`?)"
  hint="run: claude-proxy-rs mcp-proxy validate to check config"
```

The `did you mean` hint is cargo/rustc pattern — essential for TOML typos.

## 3. Bootstrap UX

### The First-Run Problem

The hardest UX question in the MCP context filter is: **how does a user know which tools they actually use before they've set up the allowlist?**

Option A: **Require manual inspection.** User reads 40 tool descriptions and guesses. Too much friction; most users will set allow = [] for everything they don't care about and miss tools they do use.

Option B: **Start permissive, tighten over time.** First-run allowlist = all tools. Then observe. Then prune. This is safe but defeats the purpose — the full 42.6k token hit continues until the user manually prunes.

Option C: **Parse past Claude Code session history.** The requirements mention this as a rabbit hole. The actual data source would be Claude Code's conversation logs or MCP call history. This is the right long-term answer but needs access to session transcript data.

Option D: **Observational dry-run proxy.** Run the proxy in passthrough mode (no filtering) for N days, logging every tool call, then auto-generate the allowlist from the call log. This is achievable without transcript access.

**Recommended bootstrap sequence:**

```
Step 1: init (day 0)
  claude-proxy-rs mcp-proxy init
  → Reads Claude Code's ~/.claude/settings.json for registered servers
  → Creates mcp-proxy.toml with all tools allowed + call-count logging enabled
  → Prints: "Proxy running in observe mode. Run 'mcp-proxy analyze' in 7 days."

Step 2: observe (days 0–7)
  Proxy runs transparently. Tool calls are logged to ~/.claude/mcp-proxy-calls.log.

Step 3: analyze (day 7)
  claude-proxy-rs mcp-proxy analyze --min-calls 1 --days 7
  → Reads mcp-proxy-calls.log
  → Prints tool call frequency table
  → Offers to rewrite mcp-proxy.toml with observed tools only
  → "28 tools never called in 7 days (saved ~18,400 tokens). Apply? [Y/n]"

Step 4: adopt
  → User confirms → allowlist tightened → next session starts with filtered context
```

This gives the user **evidence before they make any filtering decisions**. It also builds trust: they see what was dropped and why before it's dropped.

**First-run stdout (step 1):**
```
mcp-proxy init: found 14 registered MCP servers
  slack       38 tools  ~9,500 tokens
  atlassian   42 tools  ~13,000 tokens
  [... 12 more ...]
Total: 187 tools, ~42,600 tokens (21% of 200k context window)

Running in OBSERVE mode for 7 days.
Config written to: ~/.claude/mcp-proxy.toml
Run 'mcp-proxy analyze' after 7 days to generate your allowlist.
```

This number (`21% of context window`) anchors the user's understanding of what the problem costs and what filtering can save.

## 4. Observability UX

### What to Surface, What to Suppress

The existing claude-proxy-rs dashboard surfaces: requests/min, provider split, compression ratio, tokens before/after, error table, recent requests. The MCP context filter adds a new observable dimension: **tools filtered at session start**.

The right observability level for a local developer tool is:
- **High signal when something changes** (new session, allowlist updated, upstream server degraded)
- **Silent on the steady state** (same tool count, same token savings, proxy healthy)

This is the opposite of most monitoring tools, which alert on degradation and are silent otherwise. For this tool, silence means "working as expected." Noise means "something to look at."

### Metrics to Add to Dashboard

Add a new "MCP Filter" section to the existing `/dashboard` page:

| Metric | Why | Update frequency |
|--------|-----|-----------------|
| Tokens before filtering (per session) | Anchors the savings claim | Per session start |
| Tokens after filtering (per session) | Shows the actual reduction | Per session start |
| % reduction | The headline number | Per session start |
| Tools blocked this session | Sanity check: expected N tools blocked | Per session start |
| Tools called this session (by tool name) | Feeds back into allowlist tuning | Per tool call |
| Upstream server health (per server) | Surface degradation before user notices | Continuous |

### The "Quiet Unless Interesting" Log Level Strategy

```
# Default (INFO) — only log when state changes:
2026-07-02T14:00:01Z INFO  mcp_filter: session start
  slack filtered=30/38 tokens=1800 (was 9500, saved 81%)
  atlassian filtered=31/42 tokens=3200 (was 13000, saved 75%)

# WARN — log when something is degraded:
2026-07-02T14:32:01Z WARN  mcp_filter: tool called but not in allowlist
  tool=slack_create_canvas server=slack
  hint="add to [servers.slack] allow list in mcp-proxy.toml"

# DEBUG (LOG_LEVEL=debug) — per-call trace for troubleshooting:
2026-07-02T14:05:33Z DEBUG mcp_filter: tools/list request
  server=slack upstream_ms=43 tools_returned=8 tools_filtered=30
```

The WARN on "tool called but not in allowlist" is important: it's the signal that tells the user "Claude tried to use a tool you filtered." This should NOT be an ERROR — it's not a bug, it's the filter doing its job — but it should be visible so the user can decide whether to add the tool.

### What Not to Log

Never log:
- OAuth tokens or session credentials passed through the proxy
- Tool call arguments (may contain PII, credentials, or confidential data)
- Every tool call at INFO level (noise; use DEBUG)
- Session-start tool lists at INFO (one summary line is enough; full lists at DEBUG)

## 5. Mental Model

### The Core Model That Must Be Stable

The user's mental model should be: **"The proxy is a lens. It shows Claude a subset of my tools. I control the lens via the allowlist."**

This model is stable when:
- Claude can call every tool in the allowlist
- Claude cannot call tools not in the allowlist (and gets a clear error if it tries)
- The proxy logs tell the user exactly which lens was applied this session
- Editing `mcp-proxy.toml` and restarting Claude takes effect immediately

### The Failure Mode That Breaks the Mental Model

**Most dangerous failure mode: a tool silently disappears because the upstream server changed.**

Upstream MCP servers can rename, deprecate, or add tools without notice. If Slack adds `slack_create_canvas_v2` and the user has `slack_create_canvas` in their allowlist, the old tool disappears silently. Claude can no longer create canvases. The user's debugging path is: "Is Claude broken? Is Slack broken? Did something update?"

This is the `hub` failure mode from the git-ssh-fallback research — when the proxy is transparent, users lose the ability to reason about whether a failure is the tool's fault, the proxy's fault, or Claude's fault.

**Mitigations that preserve the mental model:**

1. **Log tool schema diffs at session start.** If a tool in the allowlist is no longer present in the upstream `tools/list`, log a WARN:
   ```
   WARN mcp_filter: allowed tool no longer found in upstream
     tool=slack_create_canvas server=slack
     hint="tool may have been renamed or removed; run 'mcp-proxy sync' to update"
   ```

2. **Surface "allowlist staleness" in the dashboard.** If any allowed tool hasn't been seen in upstream for N days, mark it as "possibly removed." This gives the user a visual signal without blocking them.

3. **The `mcp-proxy sync` command.** Compares current allowlist against current upstream `tools/list`, reports additions and removals, offers to update the config. This gives the user a way to actively reconcile the mental model.

### What Confuses Users the Most (Anti-patterns)

**Anti-pattern A: Proxy runs but has no effect.**
If `mcp_proxy_enabled = false` or no servers are configured in `mcp-proxy.toml`, the proxy is running but doing nothing. The user may assume it's filtering when it's not. Mitigation: startup log line must say "0 servers configured — running in passthrough mode" so the user knows.

**Anti-pattern B: Some servers filtered, others not.**
If Slack is filtered but Google Drive is not (not configured), Claude still loads Google Drive's full schema. The user sees partial savings and wonders why the remaining tools are still expensive. Mitigation: `mcp-proxy init` should offer to configure all servers, not just the ones the user asks about.

**Anti-pattern C: Restart not required.**
If the user edits `mcp-proxy.toml` and the running proxy doesn't pick it up (since claude-proxy-rs currently has no hot-reload), the user will be confused when their changes don't take effect. Mitigation: the config file's header comment should say "restart proxy to apply changes" and the dashboard should show "config last loaded: [timestamp]."

## 6. Job-to-be-Done

### Functional Job: Reduce context usage

The functional job is **done** when the user sees ≤10k tokens consumed by MCP schemas at session start instead of 42.6k. Success is measurable, observable, and binary on any given session.

Design implications:
- The proxy must measure and surface the before/after token counts on every session start
- The measurement must be in units the user already understands: "21% of context window" not "42,600 tokens"
- A regression (token count goes back up because a new server was added) should be visible immediately

**The functional job has a failure mode that's hard to detect**: a user disables a server entirely (the current workaround, per the requirements) instead of configuring it in the proxy. This means they're leaving useful tools on the table. The bootstrap UX (section 3) and observability (section 4) are the mitigation.

### Emotional Job: Feel in control of Claude's context

This is the more nuanced job. "Control" here has two dimensions:

**Dimension 1 — Legibility.** The user should be able to look at the config and know exactly what Claude can see. A good mcp-proxy.toml is a source of truth for "what tools does Claude have right now?" The token-math comments (section 1) serve this: the config file is the audit log of the user's choices.

**Dimension 2 — Confidence.** The user should not worry that filtering broke something. The WARN log on "tool not in allowlist was called" and the "possibly removed" dashboard staleness flag serve this: they surface the cases where something might have gone wrong without requiring the user to audit logs constantly.

The emotional job is broken when:
- The user doesn't know if the proxy is working (anti-pattern A above)
- Claude fails on a task and the user can't tell if filtering caused it
- The user has to restart Claude to pick up config changes and discovers this via a confusing failure rather than a clear instruction

**The right emotional signal at session start** (not in logs — visible to the user):
The proxy should not produce any user-visible output on the success path. The user's evidence that filtering worked is that their session starts faster and Claude's reasoning quality is better. If the user wants to verify, they check the dashboard. No news is good news.

## Design Recommendations Summary

### Config File

| Decision | Recommendation |
|----------|----------------|
| Config format | TOML with inline token-math comments per server |
| List semantics | Allowlist (allow =), not denylist |
| Server grouping | One [servers.NAME] block per server, alphabetical |
| Auto-generate | Yes — init command writes conservative first draft from 7-day observation |
| Hot-reload | No (keep it simple) — but config header must say "restart to apply" |

### Error UX

| Scenario | Claude sees | Log says |
|----------|-------------|----------|
| Upstream server down | MCP error with server name + timeout | WARN with consecutive_failure count + diagnostic hint |
| Tool not in allowlist (called directly) | tool_not_found with allowlist_hint | WARN once per tool per session |
| Config parse error at startup | Proxy doesn't start | ERROR with file:line and "did you mean" suggestion |
| Tool removed from upstream | Transparent (Claude just can't call it) | WARN on session start with mcp-proxy sync hint |

### Bootstrap Sequence

1. `mcp-proxy init` — discover servers from settings.json, write observe-mode config, print token cost summary
2. Observe 7 days — log every tool call to mcp-proxy-calls.log
3. `mcp-proxy analyze` — print frequency table, offer to rewrite config
4. User confirms — allowlist tightened, restart Claude

### Observability

- Dashboard: add MCP Filter section with per-server tokens before/after, % reduction, tools blocked, tools called
- Log level default: INFO for state changes, WARN for degradation, DEBUG for per-call trace
- Never log: tokens, session credentials, tool call arguments

### Mental Model Preservation

| Anti-pattern | Mitigation |
|--------------|------------|
| Proxy has no effect (misconfigured) | Startup log must state "0 servers configured — passthrough mode" |
| Tool silently removed from upstream | WARN on session start when allowed tool not found; `mcp-proxy sync` command |
| User edits config, no effect | Config header comment: "restart proxy to apply changes"; dashboard shows config loaded timestamp |
| Partial filtering (some servers unconfigured) | `mcp-proxy init` offers to configure all registered servers |
