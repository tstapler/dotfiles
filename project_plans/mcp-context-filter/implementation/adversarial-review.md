# Adversarial Review: mcp-context-filter
Date: 2026-07-02
Verdict: CONCERNS

## Blockers

- [x] **[RESOLVED] Axum version conflict has no fallback plan** — plan.md now has Task 1.1.1.0 as an explicit GO/NO-GO GATE that must complete before Task 1.1.1.1 and blocks ALL stories. The Unresolved Questions section contains a full decision tree with three ranked options (vendor + patch via `[patch.crates-io]` preferred; pin workspace to 0.7 acceptable; separate Cargo workspace as fallback) and requires documenting the chosen option as an addendum to ADR-001 before proceeding.

- [x] **[RESOLVED] Phase 3 ships BM25-only by default (34% recall)** — Epic 3.3 now describes fastembed as default-on: `[features] default = ["semantic-search"]` so a plain `cargo build --bin mcp-proxy` includes fastembed. Opt-out is via `--no-default-features`. Story 3.3.1 AC 2 adds WARN on BM25-only startup (`~34% recall on semantic queries`) and a WARN on any `search_tools` call returning 0 results in BM25-only mode. Pattern Decisions table confirms the chosen approach as "fastembed default-on; opt-out via `--no-default-features`".

## Concerns

- [ ] **Multi-process log file contention — `mcp-proxy-calls.log` will corrupt under concurrent writes** — The architecture uses one stdio subprocess per server (14 processes for 14 servers). Story 1.7.2 flushes each process's call counters to `~/.claude/mcp-proxy-calls.log` at stdin EOF, with no file locking. Fourteen processes appending JSONL concurrently without advisory locking (`flock`) will interleave partial writes, producing unparseable log lines. The `analyze` subcommand depends on this file; corrupted log data produces wrong allowlist recommendations. Recommendation: either use a per-server log file (`mcp-proxy-calls-<server>.log`) aggregated by `analyze`, or implement O_APPEND writes (which are atomic up to PIPE_BUF on macOS) and validate atomicity with a multi-process test.

- [ ] **Dashboard metrics IPC is entirely unspecified** — Story 2.3.1 Task says "expose via a shared metrics file or IPC" but does not choose between the two. A shared metrics file has the same multi-process write contention problem as the call log. IPC (e.g., Unix domain socket, shared memory) requires designing a protocol. With 14 concurrent proxy processes all trying to update the same dashboard, this is non-trivial. Recommendation: specify the mechanism before Phase 2 begins. A simple approach is per-server Prometheus text files in `~/.claude/mcp-proxy-metrics/` that the dashboard aggregates on read — atomic rename swap avoids contention.

- [ ] **`mcp-proxy init` generates configs for admin-provisioned servers that cannot authenticate** — Story 1.8.2 reads `~/.claude/settings.json` for MCP server URLs (including Slack, Atlassian Rovo) and writes proxy configs with `allow = []`. These generated configs look valid but the proxy cannot authenticate to admin-provisioned cloud connectors without user-supplied API keys (confirmed in pitfalls.md Risk Area 1 and the pattern-decisions table). A user who runs `init` and follows the instructions will get a proxy that fails every upstream connection silently. Recommendation: `init` must classify discovered servers as `admin-provisioned` (no `upstream_url` the proxy can use directly) vs. `user-accessible` (where the user has credentials), emit a commented-out stub for admin-provisioned entries with an explicit `# requires: api_key = "..."` placeholder, and print a prominent warning in the summary table.

- [ ] **Startup liveness check (Story 1.2.2) causes all-or-nothing proxy failure** — AC 2 returns `Err` if any upstream URL is unreachable at startup. With 14 servers, one flaky server causes the entire `mcp-proxy` process to refuse to start, blocking ALL tools for the session. This contradicts the resilience goal and is worse than the current baseline (which at least loads the working servers). Recommendation: change startup validation to WARN (not fail) for unreachable upstreams; return tools from reachable servers and surface a degraded-mode message. Reserve fail-fast for malformed TOML or missing required fields.

- [ ] **Call log flushed only on clean stdin EOF — crash loses session data needed by `analyze`** — Story 1.7.2 uses `DashMap` + `AtomicU64` in memory and flushes on stdin EOF. If Claude Code is killed (SIGKILL, crash, OS reboot), the per-session call counts are lost. Over weeks of use, this degrades the quality of allowlist recommendations from `analyze`. Recommendation: write call log entries incrementally (every N calls or every K seconds using a background Tokio task) rather than only at shutdown. Use O_APPEND for atomic line appends.

- [ ] **Performance SLO (≤200ms overhead, Phase 1) has no validation story** — Requirements specify "Tools/list response latency ≤200ms overhead vs. direct server call (Phase 1)." No story measures or gate-checks this. Without a benchmark story, the SLO is aspirational text with no enforcement. Recommendation: add a Story 1.4.4 with an acceptance criterion: given a local upstream stub responding in <5ms, `tools/list` via proxy completes in <210ms (p99) over 100 iterations. This must be checked before Phase 2 begins.

- [ ] **Tokenizer model selection is unresolved but affects the headline success metric** — Story 1.7.1 hardcodes `tiktoken-rs cl100k_base`, but Unresolved Questions flag that Claude Code may use `o200k_base`. If the tokenizer is wrong, all "tokens saved" numbers in the INFO log and metrics are incorrect. The requirements' primary success criterion (>75% reduction from 42.6k to ≤10k) could pass or fail depending on which tokenizer is used. Recommendation: empirically measure the baseline with both tokenizers against Claude Code's actual context consumption before shipping Phase 1; lock the choice in ADR-008. This must be resolved before Story 1.7.1 ships, not after.

- [ ] **No `mcp-proxy refresh` command — only `sync` (diff), no cache invalidation** — pitfalls.md Risk Area 5 explicitly calls for a `mcp-proxy refresh` subcommand that "clears the schema cache and re-fetches all upstream tools/list responses." The plan has `sync` (Story 1.8.4) which prints a diff but does NOT flush or invalidate the SchemaCache. Users who observe schema drift between a `sync` run and the next proxy session have no way to force a cache refresh short of restarting the proxy. Recommendation: Story 1.8.4 should include a `--apply` flag that clears the in-memory cache via IPC or writes a cache-bust file read at proxy startup; or add a separate `refresh` story.

- [ ] **Transition from observe mode to filter mode requires silent manual TOML editing** — The AllowListBootstrap workflow (init → observe → analyze → filter) ends with `analyze` rewriting the `allow` list in `mcp-proxy.toml`. But switching from `dry_run = true` to `dry_run = false` requires the user to manually edit the global config. No story covers this transition, and there is no `mcp-proxy enable-filter` command. A user running `analyze` will see the allowlist updated but remain in dry-run mode until they find and edit the right TOML field. Recommendation: `analyze` should prompt "Apply changes and enable filtering? [Y/n]" and toggle `dry_run = false` atomically with the allowlist update.

- [ ] **`ENABLE_TOOL_SEARCH` interaction with `ANTHROPIC_BASE_URL` unverified for Phase 3** — pitfalls.md Risk Area 3 confirms that Claude Code disables its built-in tool search when `ANTHROPIC_BASE_URL` points to a non-first-party host. If the user already uses claude-proxy-rs (which sets `ANTHROPIC_BASE_URL`), Claude Code's native tool discovery is already disabled. Phase 3's `search_tools` meta-tool becomes the ONLY discovery path. No story validates that `search_tools` works end-to-end when `ANTHROPIC_BASE_URL` is set to claude-proxy-rs's endpoint. Recommendation: add an integration test (Story 3.1.3) that runs the full Phase 3 proxy with `ANTHROPIC_BASE_URL` set and confirms `search_tools` returns correct results.

## Minors

- Proxy advertises `"tools":{"listChanged":true}` in its `InitializeResult` (Story 1.3.1 AC 1) despite knowing Claude Code has a confirmed bug where it ignores `notifications/tools/list_changed` (Issue #13646). This is misleading and may cause unexpected behavior if Claude Code fixes the bug in a future version while Phase 3's fixed-K model assumes it won't. Consider advertising `listChanged: false` and removing Story 3.2.3 entirely, or conditionally advertising it via a config flag.

- `bm25 = "0.1"` is an extremely early crate version with unknown maintenance trajectory. No crate health evaluation (weekly downloads, last commit, open issues) is mentioned in the plan or research. If this crate is abandoned after Phase 3 ships, upgrading BM25 requires a rewrite. Consider evaluating `tantivy` (mature, BM25-capable full-text search library) as an alternative, or implement BM25 inline (~150 lines of Rust) to eliminate the dependency entirely.

- `alwaysLoad` flag is absent from generated configs. pitfalls.md Risk Area 3 notes Claude Code may defer tool discovery if `alwaysLoad` is not set, making tools appear unavailable until the user searches. Story 1.9.1's `print-config` output should set `alwaysLoad: true` for all proxied servers.

- The `initialized` notification (sent by client after `initialize` succeeds per MCP spec 2025-06-18) has no story for passthrough from Claude Code to the upstream server. If the upstream requires this notification to complete its own initialization sequence, skipping it may cause silent upstream failures. Story 1.3.2's "pass-through handler" should explicitly list `notifications/initialized` as a covered method in its acceptance criteria.

- `mcp-proxy init` only discovers `upstream_url` entries from `settings.json` (Story 1.8.2). Stdio-based upstream MCP servers (referenced via `upstream_command` in `McpProxyConfig`) have no discovery path through `init`. If the user has local stdio MCP servers they want to proxy, they must write their `[servers]` blocks by hand. The `init` summary table should note stdio servers as "manual configuration required."
