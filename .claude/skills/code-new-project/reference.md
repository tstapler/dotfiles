# Reference: Personal App Stack (2026)

Condensed from `logseq/pages/Personal App Stack (2026).md` in the personal-wiki repo, itself the synthesis of a 7-subagent research pass verified against live 2026 web sources on 2026-07-03. Re-verify pricing/lifecycle claims if this skill is used long after that date — cloud pricing and product lifecycles (see the App Runner and PlanetScale examples below) change fast.

## Default Stack

| Layer | Default | Why |
|---|---|---|
| Frontend | **Angular** (current LTS, zoneless + signals + Vitest) | Opinionated/structured framework matching Tyler's stated preference; zoneless+Vitest confirmed production-ready as of Angular 21 |
| Frontend (KMP-sharing exception) | **Compose Multiplatform / Web (Wasm)** | Only when sharing real Kotlin logic with an existing KMP mobile/desktop app is the point. Beta as of 2026 (promoted from alpha) — accept weaker SEO/accessibility/observability tooling than Angular in exchange for code-sharing. |
| Backend | **Rust + Axum + sqlx** | Confirmed 2026 community-default Rust web framework. No framework-level architectural guardrails (unlike Spring Boot) — impose structure via a fixed module-per-layer convention from day one. Real dev-velocity tax expected for the first 2-3 months if the team is new to Rust. |
| Backend (fallback) | **Quarkus + Kotlin** | If Rust's lack of guardrails proves too unstructured in practice, or a specific project needs faster initial dev velocity over long-term cost/perf. Native-image cold starts confirmed <10ms with full Kotlin coroutine support (watch: Quarkus 4, beta ~Sept 2026, floated dropping default coroutine support — verify before relying on this path). |
| RPC — unary/server-streaming | **Connect-RPC** (connectrpc.com), Protobuf schemas in `proto/` | Callable directly from the browser via `@connectrpc/connect-web` (mature, well-documented) with no proxy — speaks the Connect protocol, native gRPC, and gRPC-Web from one service definition. **Rust server side is experimental**: `connectrpc-axum` and `axum-connect` both exist and both integrate natively with Axum, but `connectrpc-axum`'s own maintainers say it's "not yet recommended for production use." Scaffolded but explicitly flagged (`CONNECT_RPC_NOTE.md`) rather than treated as validated. |
| RPC — bidirectional streaming | **WebSocket** (plain Axum + `prost`), not Connect-RPC | As of early 2026 no stable browser ships true HTTP/2 duplex fetch (`duplex: "full"`), so a browser client cannot send and receive concurrently on one Connect/gRPC-Web stream. Reserve WebSocket for genuinely bidirectional exchanges (chat, live collaboration); unary/server-streaming should still go through Connect-RPC. This half of the RPC layer is mature/stable tooling only — smoke-tested as part of the scaffold. |
| Compute | **GCP Cloud Run** (services + jobs) | Confirmed true native $0-idle scale-to-zero (default CPU-during-requests billing + min-instances=0). Best container portability of any option surveyed (plain OCI, Knative-based). Jobs mode covers batch/cron work with no idle concept at all. |
| Compute (secondary) | **Fly.io Machines** | Use when avoiding GCP specifically matters. Keep state in Neon, not a Fly volume — volumes bill even when the Machine is stopped ($0.15/GB/mo). Free tier reportedly gone as of 2026. |
| Compute — ruled out | AWS App Runner, AWS Fargate | App Runner stopped accepting new customers April 30, 2026 (migrate to ECS Express Mode instead). Fargate has no native scale-to-zero primitive — requires hand-rolled autoscaling-to-0. |
| Data storage | **Neon** (serverless Postgres) | True default scale-to-zero (5-min idle timeout, per-request compute suspend), strong free tier (100 CU-hours/mo, 0.5GB storage), got cheaper in 2025 post-Databricks-acquisition. Standard Postgres wire protocol — no lock-in. |
| Data storage — ruled out | PlanetScale | Free/Hobby tier killed in 2024; all databases now require a paid plan starting at $39/mo. Fails the cost requirement outright. |
| Blob storage | **Cloudflare R2** | Zero egress fees, always (vs. S3's 100GB/12-month free window then $0.09/GB). S3-API-compatible — low switching cost. 10GB storage + 1M Class A + 10M Class B operations free monthly. |
| Build system | **Gradle** (if any Kotlin/KMP present) + **Cargo workspace** (Rust) | No Bazel for now — `rules_kotlin` is actively maintained in 2026 but BUILD-file authorship/maintenance still costs more than it saves at solo/small-team scale. Revisit only if team size or build-time pain actually justifies it. |
| Quality — Rust | `cargo-nextest` + `clippy` (workspace lints) + `rustfmt` + `cargo-deny` + `cargo-llvm-cov` | `cargo-deny` supersedes `cargo-audit` (superset: also license compliance, banned crates). Scope Miri only to modules with actual `unsafe` code, not workspace-wide. |
| Quality — frontend | Angular's **Vitest** builder + **Playwright** E2E | Vitest confirmed production-ready as of Angular v21 — resolves the old Karma/Jasmine churn risk. |
| Quality — cross-cutting | **Lefthook** (polyglot pre-commit) + one pinned containerized CI image | Build the container (pins JDK/Gradle if present, Rust toolchain via `rust-toolchain.toml`, Node version) *before* wiring up individual linters — that's the actual lever for local/CI parity, not any single linter choice. |
| IaC | **OpenTofu** (default) or **Pulumi via Java SDK, written in Kotlin** (ergonomics-first alternative) | Terraform is IBM-owned post-$6.4B acquisition, under BSL 1.1. OpenTofu ~12% practitioner adoption in 2026, ahead of Terraform on some features (native state encryption). Pulumi's Kotlin support is unofficial/community (`pulumi-kotlin`) but works fine in practice since Kotlin calls Java APIs natively — use it only when authoring real reusable infra classes matters more than ecosystem size for a given project. |
| CI/CD | **GitHub Actions**, path-filtered matrix jobs | Don't rebuild/retest unrelated languages on every push — use `dorny/paths-filter` or equivalent to scope jobs to `backend/**` vs `frontend/**`. |
| Multi-environment | Neon: **single project, branch per environment** (`main`=prod, persistent `staging` branch). GCP: **separate project per environment**. OpenTofu: **separate `environments/{staging,prod}/` directories**, each with its own backend/state, calling a shared `modules/app/` module — not workspaces, not a single parameterized module. | Deliberately mismatched, not uniform: put the "cheap/shared" option where the tool is designed for it (Neon branching — copy-on-write, autosuspend) and hard boundaries where a mistake is expensive (GCP IAM/project, OpenTofu state). Workspaces are a well-documented Terraform footgun for env separation (forgetting the active workspace is one of the most-cited real incidents) — confirmed still true for OpenTofu in 2026. GCP projects are free to create; splitting a single-project setup later means new Cloud Run URLs and re-pointed IAM, so get this right from day one. |
| CD pipeline | **Workload Identity Federation** (`google-github-actions/auth@v3`, keyless OIDC) + `docker buildx build --push` to Artifact Registry + **`tofu apply -var container_image=...`** (not `gcloud run deploy`, not Cloud Run's native GitHub/Cloud-Build integration) | Confirmed current (June 2026 docs). Letting `gcloud run deploy` or Cloud Run's native Cloud-Build-triggered CD touch the same resource OpenTofu manages causes permanent drift on the next `tofu plan` — a well-established Terraform/OpenTofu community lesson, not stack-specific. Rollback = revert the `container_image` value through the same `tofu apply` path, not manual `gcloud run services update-traffic` (except as an emergency break-glass, followed by a reconciling apply). Staging auto-deploys on merge to `main`; prod is a manual `workflow_dispatch` promotion of an already-validated image tag. |
| Local dev | **Neon Local** (official Docker proxy, ephemeral branch per `docker compose up`) | Confirmed to exist and be actively documented in 2026 (`neondatabase/neon_local`) — this was flagged uncertain in the first research pass and turned out to be a strong, real answer: gives docker-compose-style "fresh DB every time" ergonomics while exercising Neon's actual pooler/branching behavior, unlike a generic local Postgres container that silently diverges from prod. Trade-off: still needs network access (not a true offline emulator) — acceptable for a solo dev who's usually online. |
| Migrations | **sqlx-cli** conventions (`migrations/`, up-only, roll-forward not down-migrations) + a standalone **`migrator` binary crate** calling `sqlx::migrate!()` programmatically as the Cloud Run **Job** entrypoint | sqlx-cli is the natural default given sqlx is already the query layer. Migrations must use Neon's **direct** (non-pooled) connection string, not the pooled one the app uses at runtime — Neon's pooler runs PgBouncer in transaction mode, which doesn't support session-level `SQL PREPARE`/locking (confirmed real via Neon's own community forum and a Prisma GitHub issue). Running migrations at app-boot time is a known anti-pattern with Cloud Run's scale-to-zero (concurrent cold-starts can race to apply migrations) — a dedicated Job that the deploy pipeline waits on before promoting a new revision avoids this. |
| Secrets (dev) | **direnv + `.env`** baseline; upgrade to **1Password CLI (`op run`)** if already a subscriber | `.env` + direnv is near-zero friction and the universal baseline. `op run` is a strict upgrade when available — confirmed current passwordless desktop-app integration in 2026 (sign in once, biometric unlock after) — because the real solo-dev risk is secret *sprawl over time* (old plaintext copies on old laptops/backups), not access control (nobody else has your laptop). CI: GitHub Actions secrets for the few genuinely static tokens (`NEON_API_KEY`, `CLOUDFLARE_API_TOKEN`); WIF removes the need for any long-lived GCP credential in GitHub at all. |

## Firebase Migration Addendum

If replacing an existing Firebase app rather than building greenfield:

- **Same-niche bundled BaaS considered and rejected**: Supabase (compute pauses-after-inactivity, weaker than Neon's true per-request suspend — only reconsider if the bundle convenience outweighs that), Appwrite (self-hosted, more deployment engineering to keep scale-to-zero), PocketBase (ruled out — not recommended for production-critical apps, single-writer SQLite, no horizontal scaling), Convex (elegant reactive model but TypeScript-native, conflicts with the Rust+Axum backend decision).
- **Auth**: **Ory Kratos** over Zitadel. Apache 2.0 (vs. Zitadel's AGPL 3.0 — copyleft asterisk if ever offered as a service to others) and headless model matches the "own the pieces" philosophy — more login screens in the same Angular app rather than an embedded third-party UI. Reconsider Zitadel if dev speed matters more than architectural purity for a specific migration — its Cloud Run scale-to-zero story is more explicitly documented and it ships a turnkey login UI.
- **Realtime/offline sync** (only if the app genuinely uses Firestore-style realtime listeners, not just plain CRUD): **PowerSync** — confirmed most mature/production-recommended as of 2026, full bidirectional Postgres↔client-SQLite sync, official Kotlin Multiplatform SDK. ElectricSQL is the read-path-only alternative (writes go through your own API, which Axum already provides) but less mature — rebuilt from scratch in 2024.
- **No universal off-the-shelf Firebase-SDK-compatible shim exists.** The realistic migration path is composing Neon + R2 + Ory Kratos + (optionally) PowerSync from already-decided pieces, not adopting a new bundled vendor.

## Dominant Trade-off (for judgment calls not covered above)

Every layer where "give me guardrails" (Angular, Spring-Boot-like structure, HCL's ecosystem) collided with "minimize cost and lock-in," the research consistently favored the smaller-but-purpose-built option over the biggest incumbent — Cloud Run over AWS-native PaaS, Neon over DynamoDB, OpenTofu over Terraform. When a new decision isn't covered by this reference, default to the same bias: composable/portable over bundled/incumbent, unless the project has a specific reason (team size, real revenue, deadline pressure) to trade that away.
